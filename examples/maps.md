## Что такое карта (`map`)

Карта (`map[K]V`) — это встроенная хеш-таблица в Go, обеспечивающая неупорядоченную коллекцию пар "ключ-значение", где каждый ключ уникален.

```go
m := make(map[string]int)
m["age"] = 30
```

- Ключи должны быть **сравнимы** (`==`, `!=`).
- Значения могут быть любого типа.
- `nil`-карта не поддерживает запись.
- Карты — **ссылочные типы**.

> ⚠️ Нельзя использовать `map`, `slice`, `func` в качестве ключа — они не сравнимы.

## Внутреннее устройство карт

Карта (`map`) в Go — это хеш-таблица, реализованная с использованием открытой адресации и динамического масштабирования. Её устройство тонко оптимизировано для производительности и безопасности.

### Структура `hmap` (заголовок карты)

```go
type hmap struct {
    count     int        // количество элементов
    flags     uint8      // флаги состояния (итерация, запись и т.д.)
    B         uint8      // логарифм по основанию 2 от количества buckets
    noverflow uint16     // количество переполненных элементов
    hash0     uint32     // случайное зерно для хеширования
    buckets   unsafe.Pointer // указатель на массив buckets
    oldbuckets unsafe.Pointer // при росте — указывает на старые buckets
    nevacuate  uintptr    // во время роста — сколько buckets уже перемещено
    extra     *mapextra  // дополнительные поля для больших ключей/значений
}
```

- `count` — текущее количество элементов. Используется для `len(m)`.
- `flags` — битовые флаги:
  - `iterator` — активна итерация
  - `oldIterator` — итератор по `oldbuckets`
  - `growing` — идёт рост
  - `sameSizeGrow` — рост без увеличения размера (редко)
  - `evacuated` — флаги состояния evacuation
- `B` — база 2 количества buckets: `2^B` buckets.
- `noverflow` — приблизительное количество `overflow` bucket'ов. Используется для отслеживания нагрузки.
- `hash0` — случайное зерно, применяемое ко всем хешам. Защищает от атак на производительность.
- `buckets` — указатель на массив `bmap`'ов. Все buckets находятся в непрерывном блоке памяти.
- `oldbuckets` — при росте указывает на старый массив buckets. Пока не будет полностью перемещён, оба массива существуют.
- `nevacuate` — количество buckets, уже перемещённых в процессе `evacuation`.
- `extra` — указатель на структуру `mapextra`, содержащую:
  - `overflow` — список свободных `overflow` bucket'ов
  - `oldoverflow` — список `overflow` bucket'ов для `oldbuckets`
  - `nextOverflow` — указатель на следующий `overflow` bucket в пуле

### Структура `bmap` (одна корзина)

```go
type bmap struct {
    tophash [bucketCnt]uint8  // хэши первых байтов ключей
    // данные следуют inline: ключи, затем значения
    // + возможно overflow указатель
}
```

- `bucketCnt` = 8 — фиксированное количество слотов в bucket'е.
- `tophash` — массив из 8 байтов, содержащих первые байты хешей ключей. Используется для быстрого отсева несовпадающих ключей.
- После `tophash` следуют ключи, затем значения. Они хранятся **поблочно**, а не парами: сначала 8 ключей, затем 8 значений.
- Если bucket переполнен, используется `overflow`-bucket, на который указывает скрытый указатель в конце данных.
- `overflow`-указатель — это не поле структуры, а указатель на следующий `bmap`, размещённый **за данными**.

#### Память внутри `bmap`

Для `map[int64]string`:
- `tophash`: 8 байт
- Ключи: 8×8 = 64 байта
- Значения: 8×(указатель + длина + данные) → 8×(8 + 8 + [длина])
- `overflow`: 8 байт (указатель, если есть)

Данные всех полей хранятся **inline**, что минимизирует кэш-промахи.

### Механизм `tophash`

`tophash` — ключ к производительности:
1. При поиске вычисляется полный хеш ключа.
2. Берётся первый байт → `tophash`.
3. В `bucket` ищется слот, где `tophash[i] == tophash_value`.
4. Только если совпал `tophash`, сравниваются полный хеш и сам ключ.

Это позволяет быстро отсекать несовпадающие ключи без вызова `==`.

### `overflow` buckets и коллизии

- Коллизия — когда два ключа попадают в один `bucket`.
- Если в `bucket` нет свободного слота, создаётся `overflow` bucket.
- `overflow` bucket — это просто ещё один `bmap`, на который указывает `overflow`-указатель.
- Цепочка `overflow` bucket'ов может быть длинной, но при росте они перераспределяются.
- `noverflow` отслеживает их количество для принятия решения о росте.

> ⚠️ `tophash[0] < minTopHash` — специальные значения:
> - `emptyRest` — все слоты с этим и дальше пусты
> - `emptyOne` — слот пуст
> - `evacuatedEmpty`, `evacuatedX`, `evacuatedY` — флаги evacuation

### Механизм `evacuation`

При росте bucket'ы перемещаются постепенно. Процесс называется `evacuation`:
- `B` увеличивается на 1 → количество buckets удваивается.
- `oldbuckets` указывает на старые buckets.
- `nevacuate` — сколько bucket'ов уже перемещено.
- При доступе к `oldbucket` — он немедленно перемещается.

### Пул `overflow` bucket'ов

- Runtime поддерживает пул `overflow` bucket'ов.
- При создании `overflow` bucket'а он берётся из пула, если есть.
- При освобождении — возвращается в пул.
- Это уменьшает нагрузку на аллокатор.

### Выравнивание и компактность

- Все `bmap` выровнены по размеру указателя.
- Размер `bmap` кратен размеру кэш-линии (64 байта) для предотвращения false sharing.
- Ключи/значения хранятся блоками для эффективного копирования.

### Пример в памяти

Для `map[string]int` с 3 элементами в одном bucket'е:
```
[t1][t2][t3][ ][ ][ ][ ][ ]  ← tophash
[k1][k2][k3][ ][ ][ ][ ][ ]  ← ключи (3 строки)
[v1][v2][v3][ ][ ][ ][ ][ ]  ← значения (3 int)
[ptr]                       ← overflow ptr (если есть)
```

Каждый `k` — `string` (указатель + длина), `v` — `int`.

## Где и как хранятся в памяти

- Заголовок `hmap` — на стеке или куче (зависит от escape analysis).
- `buckets` — всегда на куче.
- При создании через `make` — выделяется память под начальные `buckets`.
- При росте — создаются новые `buckets`, старые остаются до завершения переноса.

Пример:
```go
func getMap() map[string]int {
    m := make(map[string]int)
    m["x"] = 1
    return m // m утекает → hmap и buckets на куче
}
```

## Как алоцируются, обновляются и очищаются

### Аллокация

- Через `make(map[K]V, hint)` — `hint` подсказывает начальный размер.
- Через литерал: `m := map[string]int{"a": 1}`

При `make`:
- Вычисляется начальное `B` (например, для `hint=10` → `B=4`, 16 buckets).
- Выделяется память под `2^B` buckets.

### Обновление

- Запись: `m["key"] = value`
- Удаление: `delete(m, "key")`
- Чтение: `value, ok := m["key"]`

При вставке:
1. Вычисляется хеш ключа.
2. Определяется `bucket`.
3. Ищется свободный слот или обновляется существующий.
4. Если заполнен → создаётся `overflow` bucket.

### Рост (growing) и evacuation

Карта растёт, когда:
- `loadFactor > 6.5`
- `B < 15` и `noverflow >= B`
- `B >= 15` и `noverflow >= 1<<15`

> `loadFactor = count / (2^B)` — среднее количество элементов на bucket.

#### Этапы роста

1. **Инициализация роста**:
   - `B` увеличивается на 1 → `2^(B+1)` новых buckets.
   - `oldbuckets` = `buckets`
   - `buckets` = указатель на новый массив buckets (в 2 раза больше).
   - `nevacuate = 0`
   - `flags |= growing`

2. **Evacuation (переселение)**:
   - Происходит **лениво** — по мере доступа к элементам.
   - При `read`/`write` в `oldbucket` → он немедленно `evacuate`-ится.
   - `evacuate` означает: все элементы из `oldbucket` перемещаются в новые buckets.


3. **Стратегии evacuation**:

   - **X и Y buckets**:
     - Новые buckets пронумерованы от `0` до `2^(B+1)-1`.
     - Старые buckets: `0` до `2^B-1`.
     - При `evacuate` bucket `i`: элементы распределяются в `i` (X) и `i + 2^B` (Y).
     - Решение: `hash & (2^B - 1)` == `i`? → X, иначе Y.


   - **Слияние при `sameSizeGrow`**:
     - Редкий случай — рост без увеличения `B`.
     - Используется при большом количестве `overflow` bucket'ов.
     - `oldbuckets` не удваиваются.
     - `evacuate`-ится в те же позиции, но с уменьшением цепочек.

4. **Завершение роста**:
   - Когда `nevacuate == 2^B` (все старые buckets перемещены):
     - `oldbuckets` = `nil`
     - `extra.oldoverflow` = `nil`
     - `flags &^= growing`
     - `extra.nextOverflow` = `nil`

#### Почему инкрементальный рост?

- Избегает долгих пауз GC.
- Позволяет работать с картой во время роста.
- `read` может обращаться к `oldbuckets`, `write` — к новым.

#### Сложность роста

- Время: `O(n)` — но амортизировано.
- Память: временно `2×` из-за двух массивов buckets.
- CPU: нагрузка распределена.

#### Пример evacuation

Пусть `B=2`, `2^B=4` buckets. Рост → `B=3`, 8 buckets.

- Старый bucket `0` → новые buckets `0` (X) и `4` (Y).
- Ключ `k` с `hash(k) & 3 == 0` → X, иначе Y.

После `evacuate` старый bucket помечается как `evacuatedEmpty`.

### Очистка

- `map` не требует ручной очистки.
- Память освобождается GC, когда на `hmap` нет ссылок.
- `buckets`, `oldbuckets` — тоже.

Чтобы явно освободить:
```go
m = nil // или
m = make(map[K]V) // новая пустая карта
```

## Примеры использования

### Создание и инициализация
```go
// Пустая карта
m1 := make(map[string]int)

// С начальным размером
m2 := make(map[string]int, 100)

// Через литерал
m3 := map[string]bool{
    "admin":  true,
    "active": false,
}
```

### Чтение и запись
```go
m["key"] = 42
value := m["key"]        // если нет — zero value
value, ok := m["key"]    // безопасное чтение
if ok { /* ключ есть */ }
```

### Удаление
```go
delete(m, "key")
```

### Итерация
```go
for key, value := range m {
    fmt.Println(key, value)
}
```

> ⚠️ Порядок итерации — случайный (для безопасности).

## Особенности и поведение

### Сравнение

- Карты нельзя сравнивать с `==`, только с `nil`.
- `==` panic при попытке сравнить две карты.

```go
if m1 == m2 {} // ошибка компиляции
if m1 == nil {} // OK
```

### Нулевые значения

- `var m map[string]int` → `m == nil`
- `len(m) == 0` для `nil` и пустой карты.
- `m["key"]` → zero value (0), но `ok == false`.
- `nil`-карту нельзя записывать: `m["x"] = 1` → panic.

### Копирование

Нет встроенной функции. Ручное копирование:
```go
copy := make(map[K]V, len(original))
for k, v := range original {
    copy[k] = v
}
```

> ⚠️ Копируются только указатели, если значения — ссылочные типы.

### Ключи

- Разрешены: `int`, `string`, `struct` (если все поля сравнимы), указатели, каналы, интерфейсы (если динамический тип сравним).
- Запрещены: `slice`, `map`, `func`.

## Corner cases

### 1. Итерация и модификация

```go
for k := range m {
    if someCond(k) {
        delete(m, k) // OK
    }
}
```

- Можно удалять текущий элемент.
- Нельзя удалять несуществующий — безопасно.
- Добавление во время итерации — **недопустимо**, может вызвать рост → panic.

### 2. `map` как значение в `map`

```go
m := make(map[string]map[string]int)
m["x"]["y"] = 1 // panic: nil map
```

Решение:
```go
if _, ok := m["x"]; !ok {
    m["x"] = make(map[string]int)
}
m["x"]["y"] = 1
```

### 3. Утечка памяти через `map`

```go
type Cache struct {
    data map[string]*Item
}

// item удаляется, но ссылка в map остаётся
// → item не будет собран GC
```

Решение: всегда удалять ключ через `delete`.

### 4. Хеширование и производительность

- Плохие хеш-функции → коллизии → `O(n)` вместо `O(1)`.
- Слишком частый рост → задержки.

## Работа runtime с картами

Runtime — это движок, управляющий всем жизненным циклом карты.

### Хеширование и `hash0`

Хеширование — критически важный компонент производительности и безопасности карт.

#### Алгоритм хеширования

- Для большинства типов — **`memhash`**: быстрая, нелинейная функция на основе мультипликативных хешей.
- Для строк на CPU с поддержкой **AES-NI** — используется **AES-based hash** (более безопасный и быстрый).
- Хеш вычисляется для **всего ключа**, включая указатели, структуры, строки.

#### Рандомизация через `hash0`


- `hash0` — случайное значение, генерируемое при старте программы.
- К нему применяется XOR к результату хеширования: `final_hash = hash(key) ^ hash0`.
- Это делает хеши **непредсказуемыми** между запусками.

> ⚠️ Защита от **атак на производительность**:
> - Злоумышленник не может подобрать ключи с одинаковыми хешами → коллизии.
> - Без `hash0` атакующий может создать O(n) запросы → denial of service.


#### Вычисление `bucket` по хешу

1. Вычисляется `hash = hash(key) ^ hash0`.
2. `bucket_index = hash & (2^B - 1)` — младшие `B` битов хеша.
3. `tophash = uint8(hash >> (sys.PtrSize*8 - 8))` — старший байт хеша.

> ⚠️ `tophash == 0` заменяется на `1`, так как `0` — зарезервировано для `empty`.


#### Особенности хеширования

- **Ключи-указатели** — хешируется адрес, а не содержимое.
- **Структуры** — хешируются все поля (если они сравнимы).
- **Интерфейсы** — хешируется динамический тип и значение.
- **Срезы** — не сравнимы, не могут быть ключами.

#### Производительность хеширования

- `memhash` — O(n) по размеру ключа, но оптимизирован для малых размеров.
- `aeshash` — ещё быстрее на поддерживаемом железе.
- Хеш кэшируется для строк? **Нет.** Вычисляется каждый раз.

#### Коллизии и хеши

- Коллизия: `hash(key1) == hash(key2)` → попадают в один bucket.
- `tophash` помогает быстро отсеять несовпадающие ключи.
- При коллизии ключи сравниваются через `==`.

> ⚠️ Удачный выбор `hash0` может разнести коллизии по разным buckets при росте.


#### Гарантии

- Хеш **стабилен** для одного запуска программы.
- Хеш **не стабилен** между запусками из-за `hash0`.
- Это не влияет на корректность, но важно для тестов и сериализации.

#### Пример

```go
m := make(map[string]int)
// hash0 = 0x12345678
// key = "age" → hash("age") = 0xABCDEF00
// final_hash = 0xABCDEF00 ^ 0x12345678 = 0xB9FBA978
// B=3 → 8 buckets → bucket_index = 0xB9FBA978 & 7 = 0
// tophash = uint8(0xB9FBA978 >> 24) = 0xB9
```

### Сборка мусора

- GC видит указатели в `buckets`.
- `hmap`, `buckets`, `oldbuckets` — корни для GC.
- При инкрементальном росте GC корректно обходит обе таблицы.

### Escape analysis

- `hmap` может быть на стеке, `buckets` — всегда на куче.
- Если `map` утекает — `hmap` перемещается в кучу.

## Потокобезопасность и синхронизация

### Гарантии Go

- **Не потокобезопасны**.
- Одновременная запись — **data race** → undefined behavior, может вызвать panic.
- Одновременное чтение — безопасно.

### Практические способы

1. **`sync.RWMutex`**:
```go
type SafeMap struct {
    mu sync.RWMutex
    m  map[string]int
}
```

2. **`sync.Map`** — оптимизирован для редких записей:
```go
var m sync.Map
m.Store("key", 42)
value, _ := m.Load("key")
```

> ⚠️ `sync.Map` — не замена `map`, используется в специфических сценариях.

## Производительность и оптимизации

### Оптимизации runtime

Runtime Go применяет множество оптимизаций для максимальной производительности карт.

#### Inlining (встраивание)

- Функции `mapaccess1`, `mapaccess2`, `mapassign`, `mapdelete` часто встраиваются компилятором.
- Это устраняет накладные расходы вызова функции.
- Особенно важно для горячих путей в коде.

#### Fast path (быстрый путь)

- Для простых случаев (например, доступ к существующему элементу) существуют оптимизированные пути.
- Минимизируется количество проверок и ветвлений.
- В `mapaccess1_faststr` и других `fast`-функциях хеш и сравнение оптимизированы под конкретные типы.

#### Пул `overflow` bucket'ов

- Runtime поддерживает централизованный пул `overflow` bucket'ов.
- При создании `overflow` bucket'а он берётся из пула, если доступен.
- При освобождении (при `evacuate` или очистке карты) — возвращается в пул.
- Это существенно уменьшает нагрузку на аллокатор и частоту вызовов `malloc`.
- Пул — глобальный для всех карт в программе.

#### Выравнивание и компактность памяти

- Все `bmap` выровнены по размеру указателя.
- Размер `bmap` кратен размеру кэш-линии (обычно 64 байта) для предотвращения `false sharing` между ядрами CPU.
- Ключи и значения хранятся **поблочно** (сначала все ключи, потом все значения), что обеспечивает:
  - Лучшую локальность при массовом доступе.
  - Возможность эффективного копирования через `memmove`.
  - Избегание выравнивания в парах `key-value`.

#### Ленивая инициализация `buckets`

- При `make(map[K]V)` без элементов `buckets = nil`.
- Первое обращение (чтение/запись) инициирует аллокацию `buckets`.
- Экономит память для карт, которые создаются, но редко используются.

#### Оптимизация для малых карт

- Использование `hint` в `make` позволяет избежать ранних ростов.
- При `hint=0` или малом `hint` используется минимальный `B`.

#### Escape analysis и стековые карты

- Маленькие карты, не утекающие за пределы функции, могут быть размещены полностью на стеке.
- `hmap` — на стеке, `buckets` — на куче, но если `buckets` не создаются (карта пуста), вся карта остаётся на стеке.
- Это ускоряет создание и уменьшает нагрузку на GC.

#### Оптимизация сборки мусора

- GC знает структуру карт и может эффективно сканировать указатели в `buckets`.
- При инкрементальном росте GC корректно обходит как `buckets`, так и `oldbuckets`, обеспечивая надёжность.
- Поколения GC не мешают работе `evacuation`.

#### CPU- и память-ориентированные оптимизации

- Выбор между `memhash` и `aeshash` происходит на основе CPU.
- `aeshash` использует аппаратные инструкции AES-NI для сверхбыстрого хеширования строк.
- Использование `tophash` позволяет минимизировать количество сравнений через `==`, которые могут быть дорогими.

#### Прочие оптимизации

- **Предвычисление `hash`**: при `mapassign` хеш вычисляется один раз и используется для поиска, сравнения и `evacuation`.
- **Ленивое обновление `nevacuate`**: инкрементируется пачками для уменьшения нагрузки на атомарные операции.
- **Оптимизация `range`**: итератор использует `bucket` и `slot` для прямого доступа, избегая поиска.

### Рекомендации

- Указывайте `hint` в `make`, если известен размер.
- Избегайте частых `delete`/`insert` в больших картах.
- Используйте `sync.Map` только при паттерне: много чтений, редкие записи.
- Не используйте `map[interface{}]interface{}` без необходимости — высокая аллокация.

## Заключение

Карты — мощный и гибкий инструмент в Go, реализованный как эффективная хеш-таблица с инкрементальным масштабированием. Понимание их внутреннего устройства, особенностей роста, поведения при гонках и работы с памятью позволяет писать надёжный и производительный код. Несмотря на простоту интерфейса, карты требуют внимания к деталям: потокобезопасности, управлению памятью и выбору подходящего способа синхронизации.

Этот конспект охватывает все ключевые аспекты работы с картами в Go 1.26.2 и может служить исчерпывающим справочником для разработчиков.
