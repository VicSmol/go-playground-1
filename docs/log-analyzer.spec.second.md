# Спецификация CLI утилиты `log-analyzer`

## Назначение

Утилита `log-analyzer` предназначена для анализа текстовых файлов логов: чтения, парсинга, агрегации по ключевым атрибутам (уровень логирования, компонент системы) и вывода топ-N записей на основе заданной метрики (частота, объем данных и т.п.).

Она может использоваться как автономная команда для диагностики систем, поиска аномалий и мониторинга состояния приложений.

## Требования

### Функциональные требования

1. **Чтение лог-файлов**
   - Поддержка чтения больших текстовых файлов (GB+) с эффективным использованием памяти.
   - Поддержка стандартного ввода (stdin) для интеграции в пайплайны Unix.
   - Возможность указать путь к файлу через аргумент командной строки.

2. **Парсинг логов**
   - Предварительно заданный формат лога (пример):
     ```
     [2026-05-14T10:20:35Z] INFO component=auth msg="User login successful" user_id=12345
     [2026-05-14T10:20:36Z] ERROR component=database msg="Query timeout" duration_ms=450
     ```
   - Извлечение следующих полей:
     - Временная метка (timestamp)
     - Уровень логирования (level): DEBUG, INFO, WARN, ERROR, FATAL
     - Компонент (component)
     - Сообщение (msg)
     - Произвольные ключ-значные пары (structured fields)

3. **Агрегация данных**
   - Агрегация по:
     - Уровню логирования (`level`)
     - Компоненту системы (`component`)
     - Комбинации `level + component`
   - Подсчет количества записей, суммарного объема данных (в байтах), средней длины сообщения.

4. **Сортировка и вывод Top-N**
   - Поддержка флага `--top N` для вывода топ-N наиболее частых комбинаций.
   - Поддержка флагов сортировки:
     - `--sort count` (по количеству)
     - `--sort bytes` (по объему)
     - `--sort avg-len` (по средней длине строки)
   - Вывод в табличном формате (текстовом или CSV).

5. **Формат вывода**
   - Поддержка флагов:
     - `--output text` — человекочитаемый формат (по умолчанию)
     - `--output csv` — CSV для экспорта
     - `--output json` — структурированный JSON

### Нефункциональные требования

1. **Производительность**
   - Минимальное потребление памяти: обработка файла построчно, без загрузки в память целиком.
   - Использование буферизации (`bufio.Scanner` с оптимальным размером буфера).
   - Эффективная работа с большими файлами (10GB+).

2. **Масштабируемость**
   - Архитектура позволяет легко добавлять новые парсеры (например, JSON-логи, Apache, Nginx).
   - Поддержка плагинов или интерфейсов для расширения.

3. **Надёжность**
   - Обработка битых строк (неполные, невалидные) с пропуском и подсчётом ошибок.
   - Отчёт об ошибках парсинга при флаге `--verbose`.

4. **Портативность**
   - Работа на всех ОС, поддерживаемых Go (Linux, macOS, Windows).
   - Скомпилированный бинарник не требует зависимостей.

## Архитектура

### Компоненты

1. **`cmd/log-analyzer/main.go`** — точка входа.
   - Использует библиотеку [`cobra`](https://github.com/spf13/cobra) для построения CLI.
   - Определяет команды и флаги.

2. **`internal/loganalyzer/parser.go`** — парсер логов.
   - Интерфейс `LogParser`:
     ```go
     type LogParser interface {
         ParseLine(line string) (*LogEntry, error)
     }
     ```
   - Реализация `SimpleTextParser` для текстового формата.
   - Возможность расширения (JSONParser и др.).

3. **`internal/loganalyzer/entry.go`** — модель данных.
   ```go
   type LogEntry struct {
       Timestamp time.Time
       Level     string
       Component string
       Message   string
       Fields    map[string]string
   }
   ```

4. **`internal/loganalyzer/aggregator.go`** — агрегация.
   - Хранит данные в виде вложенных `map[string]map[string]Stats`:
     - Внешний ключ: `level`
     - Внутренний ключ: `component`
   - `Stats` — структура с полями: `Count`, `TotalBytes`, `TotalLength`.
   - Реализует методы `Add(entry *LogEntry)` и `TopN(keyFunc func(*Stats) int64, n int) []RankedItem`.

5. **`internal/loganalyzer/output.go`** — форматировщики.
   - Интерфейс `OutputFormatter`:
     ```go
     type OutputFormatter interface {
         Format(items []RankedItem) string
     }
     ```
   - Реализации: `TextFormatter`, `CSVFormatter`, `JSONFormatter`.

### Поток выполнения

1. CLI-флаги → `main`
2. Выбор парсера по формату (`--format text|json`)
3. Открытие файла (или `os.Stdin`)
4. Построчное чтение через `bufio.Scanner`
5. Парсинг каждой строки → `LogEntry`
6. Агрегация через `Aggregator.Add()`
7. Сортировка и выбор Top-N
8. Форматирование и вывод

## CLI Интерфейс

```bash
log-analyzer [flags] <logfile>
```

### Флаги

| Флаг | Описание | По умолчанию |
|------|----------|--------------|
| `--format string` | Формат логов: `text`, `json` | `text` |
| `--top int` | Количество топовых записей для вывода | `10` |
| `--sort string` | Критерий сортировки: `count`, `bytes`, `avg-len` | `count` |
| `--output string` | Формат вывода: `text`, `csv`, `json` | `text` |
| `--group-by string` | Группировка: `level`, `component`, `both` | `both` |
| `--verbose` | Выводить статистику парсинга | `false` |

### Примеры использования

```bash
# Анализ файла, топ-10 по частоте
log-analyzer --top 10 --sort count access.log

# Топ-5 компонентов по объему данных
log-analyzer --group-by component --sort bytes --top 5 app.log

# Вывод в JSON
log-analyzer --output json --format json service.log

# Через stdin
cat system.log | log-analyzer --format text
```

## Ограничения

- Поддержка только UTF-8 кодировки.
- Формат текстовых логов фиксирован — изменения требуют правки парсера.
- Нет поддержки многопоточного парсинга (пока что).
- Максимальная длина строки ограничена буфером `bufio.Scanner` (64KB по умолчанию, можно увеличить).
