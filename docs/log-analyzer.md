# Техническое задание: CLI "log-analyzer"

## 1. Обзор

Разработать консольное приложение (CLI) на языке Go с именем `log-analyzer`, которое:
- Читает файлы логов (или stdin)
- Парсит логи в нескольких форматах
- Агрегирует записи по комбинации "уровень логирования + компонент"
- Выводит топ-N самых частых комбинаций

---

## 2. Функциональные требования

### 2.1 Входные данные

#### 2.1.1 Источник данных
- CLI должен поддерживать **два источника входных данных**:
  1. **Файл**: указывается как аргумент командной строки (путь к файлу)
  2. **Stdin**: данные читаются из стандартного потока ввода (через пайп `|`)

#### 2.1.2 Форматы логов
CLI должен **автоматически определять формат** логов и поддерживать:

| Формат | Пример | Описание |
|--------|--------|----------|
| **JSON** | `{"level":"INFO","component":"auth","message":"User logged in","timestamp":"2026-06-08T10:00:00Z"}` | Каждая строка — валидный JSON объект |
| **Syslog (RFC 3164)** | `Jun  8 10:00:00 host component[123]: [INFO] User logged in` | Стандартный syslog формат |
| **Кастомный (ключевые слова)** | `INFO auth: User logged in` | Формат `LEVEL COMPONENT: message` с разделителями ` ` и `:` |

> **Примечание**: Автоматическое определение формата должно происходить на основе структуры строки (например, наличие `{` в начале — JSON, дата/время в начале — syslog, и т.д.).

#### 2.1.3 Уровни логирования
Поддерживаются **стандартные уровни** (регистронезависимо):
- `DEBUG`
- `INFO`
- `WARN` / `WARNING`
- `ERROR`
- `FATAL` / `CRITICAL` / `EMERGENCY`

> **Примечание**: `WARN`, `WARNING` должны агрегироваться как `WARN`; `FATAL`, `CRITICAL`, `EMERGENCY` — как `FATAL`.

#### 2.1.4 Компоненты
Компонент — это идентификатор части системы, генерирующей лог:
- В JSON: поле `component`
- В syslog: имя процесса/сервиса (например, `nginx`, `auth-service`)
- В кастомном формате: слово после уровня до `:` (например, `auth` в `INFO auth: ...`)

### 2.2 Агрегация

#### 2.2.1 Группировка
- **Ключ агрегации**: `level + component` (например, `INFO:auth`, `ERROR:db`)
- Значение: **количество вхождений** данной комбинации

#### 2.2.2 Сортировка
- **По умолчанию**: по убыванию количества (`count DESC`)
- **Опция сортировки**: через флаг `--order`
  - `--order desc` (по умолчанию) — убывающая
  - `--order asc` — возрастающая
- **Тип сортировки**: лексикографическая не применяется

### 2.3 Вывод результатов

#### 2.3.1 Форматы вывода
Поддерживаются **три формата** через флаг `--output`:
- `table` (по умолчанию) — человекочитаемая ASCII-таблица
- `json` — структурированный JSON
- `csv` — CSV-формат

#### 2.3.2 Структура вывода

##### Таблица (`table`)
```
+------------+------------------+-------+
|   LEVEL    |   COMPONENT      | COUNT |
+------------+------------------+-------+
| INFO       | auth             |    42 |
| ERROR      | db               |    17 |
| WARN       | cache            |    13 |
+------------+------------------+-------+
```

##### JSON (`json`)
```json
[
  {
    "level": "INFO",
    "component": "auth",
    "count": 42
  },
  {
    "level": "ERROR",
    "component": "db",
    "count": 17
  }
]
```

##### CSV (`csv`)
```
level,component,count
INFO,auth,42
ERROR,db,17
```

#### 2.3.3 Ограничение количества строк
- **Флаг**: `--top N`
- **Значение по умолчанию**: `5`
- **Поведение**: ограничить вывод **N** первыми записями после сортировки

### 2.4 Обработка ошибок

#### 2.4.1 При чтении файла
- Если файл не найден → вывести сообщение об ошибке в `stderr` и завершиться с кодом `1`
- Если нет прав на чтение → вывести сообщение об ошибке в `stderr` и завершиться с кодом `1`
- Если файл пустой → завершиться с кодом `0` и пустым выводом (без ошибок)

#### 2.4.2 При парсинге строк
- **Строка не распознана** (не соответствует ни одному формату) → **пропустить строку и продолжить**
- **Некорректный JSON** → **пропустить строку и продолжить**
- **Парсинг не удался по другой причине** → **пропустить строку и продолжить**

> **Примечание**: CLI должен быть устойчив к "грязным" данным — не падать на битых строках.

#### 2.4.3 Коды завершения (exit codes)
- `0` — успех (даже если часть строк пропущена)
- `1` — ошибка ввода-вывода (файл не найден, нет прав и т.п.)

## 3. Нефункциональные требования

### 3.1 Архитектура
- CLI должен быть реализован как **подкоманда** в главном CLI проекта (через `cobra` или встроенный `flag`)
- Код должен быть организован по модулям:
  - `pkg/log-analyzer/parser/` — парсеры логов
  - `pkg/log-analyzer/aggregator/` — агрегатор данных
  - `pkg/log-analyzer/output/` — форматы вывода
  - `internal/log-analyzer/` — общие утилиты (например, версия CLI)
  - `cmd/main.go` — главная точка входа (регистрирует подкоманды)

### 3.2 Качество кода
- Соблюдать **Go Code Review Comments** и **Effective Go**
- Использовать **интерфейсы** (например, `LogParser`, `Formatter`)
- Покрыть код **unit-тестами** (`*_test.go` в тех же пакетах)
- Использовать **стандартные библиотеки Go** (не требовать внешних зависимостей, кроме `cobra` при необходимости)

### 3.3 Производительность
- Обработка должна быть **потоковой**: не загружать весь файл в память
- Для больших файлов использовать `bufio.Scanner`

### 3.4 Документация
- В `README.md` должен быть пример использования CLI
- В `docs/log-analyzer.md` — это техническое задание

## 4. Спецификация CLI (флаги и аргументы)

### 4.1 Позиционные аргументы
| Аргумент | Описание | Обязательный |
|----------|----------|--------------|
| `FILE` | Путь к файлу логов (если не указан, читать из stdin) | Нет |

### 4.2 Флаги
| Флаг | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `--top N` | `int` | `5` | Количество строк в выводе |
| `--order` | `string` | `"desc"` | Порядок сортировки: `asc` или `desc` |
| `--output` | `string` | `"table"` | Формат вывода: `table`, `json`, `csv` |
| `--help` | `bool` | `false` | Показать справку |
| `--version` | `bool` | `false` | Показать версию CLI |

### 4.3 Примеры использования
```bash
# Анализ файла, топ-5 (по умолчанию)
log-analyzer /var/log/app.log

# Анализ файла, топ-10, по возрастанию
log-analyzer --top 10 --order asc /var/log/app.log

# Анализ stdin, вывод в JSON
cat app.log | log-analyzer --output json

# Анализ файла с топ-3 в CSV
log-analyzer --top 3 --output csv /var/log/app.log
```

## 6. Технические детали

### 6.1 Структура агрегированных данных
```go
type LogEntry struct {
    Level     string
    Component string
    Count     int
}

type Aggregator struct {
    stats map[string]int // key: "level:component"
}

func (a *Aggregator) Add(level, component string)
func (a *Aggregator) GetStats() []LogEntry
```

### 6.2 Уровни логирования
- Нормализация: `WARN` и `WARNING` → `"WARN"`, `FATAL`, `CRITICAL`, `EMERGENCY` → `"FATAL"`
- Вывод: нормализованные значения (`INFO`, `WARN`, `ERROR`, `FATAL`, `DEBUG`)

### 6.3 Автоматическое определение формата
| Условие | Формат |
|---------|--------|
| Строка начинается с `{` | JSON |
| Строка начинается с месяца (например, `Jun`, `Feb`) | Syslog |
| Строка содержит `INFO/WARN/ERROR/FATAL/DEBUG` в начале (до `:`) | Custom |

### 6.4 Потоковая обработка
```go
scanner := bufio.NewScanner(reader)
for scanner.Scan() {
    line := scanner.Text()
    if entry, err := parser.Parse(line); err == nil {
        aggregator.Add(entry.Level, entry.Component)
    }
}
```
